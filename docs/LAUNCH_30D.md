# 30-Day Launch Execution (Claw Windows)

## Goal

- Week 1: finish product narrative + onboarding polish.
- Week 2: produce distribution assets and start public sharing.
- Week 3: convert early users into testimonials and case studies.
- Week 4: improve activation funnel and retention.

## Success Metrics

- Activation: install -> paired in Feishu within 20 minutes.
- First value: user sends first successful automation command within 1 day.
- Retention: user active in 7 days after first install.
- Referral: users sharing tutorial/repo links.

## Week 1 (Product Readiness)

Day 1
- Freeze messaging and positioning.
- Finalize README first-screen message.

Day 2
- Record clean setup walkthrough (screen recording).
- Publish quickstart GIF/video snippets.

Day 3
- Build FAQ for top setup issues.
- Add "doctor check" troubleshooting flow in docs.

Day 4
- Validate on at least 2 clean Windows machines.
- Collect setup time and failure points.

Day 5
- Improve GUI text and error prompts.
- Add one support contact path in README.

Day 6
- Prepare release package and changelog.
- Publish tagged release.

Day 7
- Internal review: activation funnel drop-off.
- Choose top 3 UX fixes for next week.

## Week 2 (Traffic + Awareness)

Day 8-9
- Publish 1 short "why this exists" post (CN + EN).
- Publish 1 tutorial post focused on Feishu onboarding.

Day 10-11
- Share in relevant OpenClaw/automation communities.
- Ask for first 10 external installs.

Day 12
- Publish "Windows no-WSL no-Docker" comparison article.

Day 13
- Collect user objections and rewrite onboarding copy.

Day 14
- Ship minor release focused on friction fixes.

## Week 3 (Conversion)

Day 15-16
- DM early users for setup feedback.
- Turn successful cases into short stories.

Day 17
- Add "first 10-minute checklist" section in README.

Day 18-19
- Produce one office-workflow demo (Feishu + local files).

Day 20
- Publish pricing/packaging hypothesis (if monetizing).

Day 21
- Release with stronger activation guidance.

## Week 4 (Retention + Scale)

Day 22-23
- Analyze common failures.
- Add specific fixes in docs and scripts.

Day 24-25
- Add "self-check + logs export" flow.
- Reduce support response time via templates.

Day 26
- Publish monthly recap with metrics and roadmap.

Day 27-28
- Prepare contributor guide and issue templates.

Day 29
- Review positioning vs competitors.

Day 30
- Decide next growth focus:
  - deeper Feishu experience
  - enterprise onboarding
  - managed hosting variant

## Distribution Checklist (Always-On)

- Keep README first 30 lines conversion-focused.
- Keep one short install video always updated.
- Keep "doctor" script output easy to share.
- Keep release notes short and user-centered.
